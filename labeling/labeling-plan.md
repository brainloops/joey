# Basketball MOT Labeling Plan

## Purpose

- Build a fast, low-friction labeling workflow for basketball clips that outputs MOT-compatible data for immediate use with `TrackEval` and benchmark-style evaluation.
- Keep manual effort focused on identity stitching/correction rather than drawing every box from scratch.

## Intended Usage

- Input:
  - extracted action-centered clips (`10s pre + 10s post` default, configurable) from scored game JSON events
  - frame folders (`img1/`) per clip
  - model pre-label detections (`det/det.txt`) generated by a pinned RF-DETR pass
- Human labeling:
  - stitch detection boxes into persistent player IDs
  - resolve ambiguous assignments
  - fill short gaps via skip/relink actions
- Output:
  - MOT-style sequence package per clip:
    - `img1/`
    - `gt/gt.txt`
    - `seqinfo.ini`
    - optional retained `det/` for auditability and relabeling

## Vault-Aware Source Conventions

- Source videos live in the Vault and are identified by a unique `vault_id` (filesystem-derived identifier).
- Every derived clip/sequence must retain source provenance via `vault_id` in name or metadata.
- Minimum provenance fields to persist per clip:
  - `vault_id`
  - source game/scoring JSON identifier
  - action type
  - action timestamp/frame anchor
  - clip start/end timestamps
- Recommended:
  - keep a sidecar metadata file per clip/sequence (for example `clip_meta.json`) so names stay readable while provenance remains lossless.

## Dataset Staging Layout (Clip-Centric)

- Organizational policy:
  - one folder per source game under a clips staging root
  - inside each game folder, one folder per extracted clip (this clip folder is the MOT sequence unit for labeling/eval)
- Suggested structure:

```text
labeling/
  clips/
    <vault_id_or_game_key>/
      <clip_id>/                # one extracted clip
        clip.mp4                # optional convenience artifact
        clip_meta.json
  sequences/
    <vault_id_or_game_key>/
      <clip_id>/                # one MOT sequence for label/eval
        img1/
        det/
        gt/
        seqinfo.ini
```

- Naming guidance:
  - derive `clip_id` deterministically from source + action window
  - example pattern:
    - `<vault_id>__<action_type>__t<start_ms>-<end_ms>`
  - avoid random UUID-only naming for primary sequence IDs (harder to debug/audit)

## MOT Compatibility Requirements

### `gt/gt.txt` (ground-truth rows)

- Use 9-column MOT17/MOT20-style GT rows:
  - `frame,id,bb_left,bb_top,bb_width,bb_height,conf,class,visibility`
- Key conventions:
  - frame index is 1-based
  - ID is persistent per real player identity
  - `conf` in GT acts as include/ignore flag (`0` ignored, non-zero active)

### Tracker/prediction compatibility

- Keep downstream exports compatible with 10-column MOT results:
  - `frame,id,bb_left,bb_top,bb_width,bb_height,conf,x,y,z`
- For 2D tracking, `x,y,z` use `-1`.

### `seqinfo.ini` requirements

- Each sequence must include:
  - `name`
  - `imDir`
  - `frameRate`
  - `seqLength`
  - `imWidth`
  - `imHeight`
  - `imExt`
- `seqLength` and image geometry must match disk contents.

## Data Contract v1 (Operational Rules)

- Canonical detector output for downstream tooling must exist at:
  - `det/det.txt`
- Optional detector audit output may also be written (recommended):
  - `det/det_rfdetr.txt`
- If both are written, keep them byte-identical for reproducibility.

- Canonical benchmark naming for TrackEval integration:
  - choose and freeze one benchmark label for internal data (for example `BasketballMOT`)
  - TrackEval GT split folder format: `<benchmark>-<split>`
  - Seqmap filename format: `<benchmark>-<split>.txt`
  - Example:
    - `.../gt/mot_challenge/BasketballMOT-train`
    - `.../gt/mot_challenge/seqmaps/BasketballMOT-train.txt`

- Sequence media normalization requirement:
  - tracking/detection stages operate on numeric frame files in `img1/` (for example `000001.jpg`)
  - if source arrives as `img1.mp4`, decode to `img1/*.jpg` before running detector/tracker
  - keep `seqinfo.ini` aligned with decoded frame count and image geometry

- File sanitation requirement (hard):
  - ignore non-numeric frame stems when enumerating frames
  - ignore/remove macOS metadata artifacts such as:
    - `__MACOSX/`
    - `._*`
  - this prevents parse failures such as `int('._000001')`

- GT-ready split policy:
  - local TrackEval runs only on splits where each sequence has:
    - `seqinfo.ini`
    - `gt/gt.txt`
  - prediction-only runs may still execute on non-public-GT splits (for example many `test` splits)

- Evaluation caching policy:
  - comparisons should reuse existing `pedestrian_summary.txt` when present
  - only rerun tracker/eval when outputs are missing or explicitly forced
  - this is required for practical iteration when slow trackers are included

## Detector-Assisted Pre-Label Pipeline

- Design intent:
  - maximize annotation speed and consistency through one reproducible pre-label pass
- Pipeline:
  1. for each scored game JSON + source video (`vault_id`), extract 50-100 action clips (or dataset-defined count)
  2. stage clips under game-level folder using deterministic `clip_id` naming
  3. extract frames to sequence `img1/`
  4. run pinned RF-DETR checkpoint/config on all frames
  5. write proposals to `det/det.txt`
  6. load `img1/` + `det/` in labeler and stitch into identities
  7. export final `gt/gt.txt` + `seqinfo.ini`
- `det/det.txt` conventions:
  - use MOT 10-column shape for consistency
  - set `id=-1` for raw detections
  - keep detector confidence in `conf`
  - set `x,y,z=-1`
- Reproducibility in `det/metadata.json`:
  - model name/version/checkpoint
  - confidence and NMS thresholds
  - resize/preprocessing config
  - code commit hash

## Primary UX: Single-Track Click-Through

- Core interaction:
  1. start at frame 1 with all unmatched detections
  2. click one detection to seed a track ID
  3. advance to next frame and show only near candidates
  4. accept top candidate quickly or click alternate when ambiguous
  5. continue until track end, then return to frame 1
  6. repeat with remaining unmatched detections
- Why this works:
  - candidate set shrinks as tracks are finalized
  - fewer conflicts over time means faster annotation later in each clip

## Candidate Matching and Ambiguity Handling

- Candidate gating (MVP):
  - IOU vs last accepted box
  - center-distance threshold (normalized by bbox scale)
  - optional constant-velocity projection
- Ambiguity behavior:
  - display ranked candidates and require explicit click on ambiguous frames
  - keep non-candidates visually suppressed

## Gap and Re-Entry Policy

- UI gap controls:
  - skip 1 frame / skip N frames
  - relink when a strong candidate reappears
  - configurable max gap for auto-suggestion/auto-end workflow behavior
- Ground-truth identity rule (hard requirement):
  - if annotator is confident it is the same real player, keep the same `track_id` across gaps/re-entry
  - do not force a new GT ID solely because a frame-gap threshold is exceeded

## Hotkeys and Ergonomics

- Constraint:
  - all primary hotkeys must be left-hand reachable (right hand stays on mouse)
- MVP defaults:
  - `Space`: accept top candidate and advance
  - `A`: previous frame / undo last step
  - `D`: next frame
  - `W`: skip frame
  - `Q`: end current track
  - `E`: start next track
  - `S`: save
  - `R`: relink/resume search mode
  - mouse click: choose alternate candidate

## Export Validation Rules

- enforce positive integer IDs and 1-based frame indices
- validate bbox width/height > 0 and in-bounds handling
- validate column count/order for each output row
- validate `seqinfo.ini` vs actual frame directory contents
- ensure each detection assignment is unique unless explicitly reassigned by user
- support reversible edits (`undo/redo`)

## Acceptance Criteria

- Annotator can complete typical 20s clips primarily with `Space` + occasional clicks.
- At least 3 clips are labeled end-to-end and exported in MOT-compatible structure.
- Exported clips pass `benchmarks/verify_layout.sh` with zero failures.
- `TrackEval` runs on exported custom sequences without conversion scripts.
